# 仙侠家族小说正文生成

## 默认工程

除非用户明确指定别的小说工程目录，否则一律使用默认共享工程：

`jiazuxiuxian_xiaoshuo_skills/jiazuxiuxian_novel-chapter-writer/assets/sample-novel-project/`

默认工程采用如下结构：

- `设定/`
- `大纲/`
- `正文/`
- `追踪/`
- `参考资料/`

其中：

- `设定/世界观/` 放世界观、修炼体系、地图、规则
- `设定/角色/` 放角色卡
- `设定/势力/` 放家族、宗门、坊市、附庸等势力文件
- `设定/关系.md` 放关系映射
- `设定/题材定位.md` 放题材卖点、对标、风格约束
- `大纲/大纲.md` 放全书卷级结构
- `大纲/卷纲_*.md` 放单卷结构
- `大纲/细纲_第NNN章.md` 放单章细纲
- `正文/第NNN章_章名.md` 放正文
- `追踪/上下文.md`、`追踪/伏笔.md`、`追踪/时间线.md`、`追踪/角色状态.md` 做连续性管理

如果当前仓库根目录存在：

`《家族修仙：这一脉，由我来立》小说策划案.md`

则把它视为当前默认项目的主策划案和外部总锚点。

## 人工二审覆盖层

保留人工二审覆盖机制，但不再使用历史项目那套目录。

如果小说工程根目录存在以下任一目录：

- `正文_人工二审/`
- `人工二次审核过后的正文/`

执行规则：

- 同章节号若人工二审目录与 `正文/` 都存在，一律优先读取人工二审稿
- 人工二审稿同时视为更高优先级的 canon 和更可靠的文风锚点
- 继续写新章时，默认仍写回 `正文/`
- 只有用户明确要求时，才把新稿直接写进人工二审目录

## 用户常见说法

- 写第 8 章
- 续写第 9 章正文
- 重写第 6 章，让族议更有压迫感
- 根据现有设定和前文生成第 10 章

## 默认字数

除非用户明确指定更短或更长的字数目标，否则默认把单章正文控制在 4000 字左右。

执行口径：

- 默认目标区间为 3800 到 4300 字
- 非特殊情况不低于 3500 字，不高于 4500 字
- 低于 3500 字时，优先补足事务、阻力、兑现和章末牵引，不要靠空泛感慨硬拖
- 高于 4500 字时，优先压缩重复解释、合并弱场景，必要时把次要信息后移到下一章
- 如果用户明确要求“3000 字短章”“5000 字大章”或其他体量，以用户要求为准

## 信息优先级

遇到信息冲突时，严格按下面优先级处理：

1. 工程内 `正文_人工二审/` 或 `人工二次审核过后的正文/` 中存在的对应章节
2. 工程内 `正文/` 中已成稿正文
3. 工程内 `大纲/细纲_第NNN章.md`
4. 工程内 `大纲/卷纲_*.md`
5. 工程内 `大纲/大纲.md`
6. 工程内 `追踪/上下文.md`、`追踪/伏笔.md`、`追踪/时间线.md`、`追踪/角色状态.md`
7. 工程内 `设定/` 下的相关文件
8. 仓库根目录的 `《家族修仙：这一脉，由我来立》小说策划案.md`
9. skill 内的 `references/current-story-state.md` 与 `references/novel-bible.md`
10. skill 内的通用写作约束和平台规则
11. 一般性仙侠、家族修仙、连载文默认推断

不要让参考设定反过来覆盖已成稿正文。

## 工作流程

### 1. 先定位上下文

先运行：

```bash
python3 scripts/novel_context.py chapter --chapter N
```

脚本会返回：

- 当前章输出文件
- 当前章优先读取的 source 文件
- 最近一章正文
- 最近三章正文列表
- 当前章细纲文件
- 全书大纲与卷纲文件位置
- `设定/`、`追踪/`、`参考资料/` 路径
- 是否启用了人工二审覆盖层
- 外部主策划案位置
- 当前工程根目录

### 2. 先读哪些文件

正文生成前，按下面顺序读取：

1. 仓库根目录的 `《家族修仙：这一脉，由我来立》小说策划案.md`，如果存在
2. 工程内 `设定/题材定位.md`
3. 工程内 `设定/关系.md`
4. 工程内与本章直接相关的 `设定/角色/*.md`
5. 工程内与本章直接相关的 `设定/势力/*.md`
6. 工程内与本章直接相关的 `设定/世界观/*.md`
7. `大纲/大纲.md`
8. 相关 `大纲/卷纲_*.md`
9. `大纲/细纲_第NNN章.md`
10. `追踪/上下文.md`
11. `追踪/伏笔.md`
12. `追踪/时间线.md`
13. `追踪/角色状态.md`
14. `references/current-story-state.md`
15. `references/novel-bible.md`
16. `references/naturalness-rules.md`
17. `references/readability-rules.md`
18. `references/qidian-editor-rules.md`
19. `references/qidian-opening-patterns.md`
20. `references/tracking-protocol.md`
21. `references/xianxia-hook-rules.md`
22. 如果存在 `references/manual-review-findings.md`，默认补读
23. 默认至少读 `references/modules/plot_logic/runtime.md`
24. 默认至少读 `references/modules/chapter_ending/runtime.md`
25. 完稿前必读 `references/modules/consistency_review/runtime.md`
26. 完稿前必读 `references/modules/anti_ai_voice/runtime.md`
27. 关系戏重时，补读 `references/modules/dialogue/runtime.md`
28. 与上一章承接紧时，补读 `references/modules/transition/runtime.md`
29. 人物状态复杂或跨章余波重时，补读 `references/modules/character_consistency/runtime.md`
30. 最近一章正文
31. 如当前章承接关系很强，再补读最近两到三章正文

如果当前章已有旧稿，必须先读旧稿，再决定是覆盖重写还是在原文基础上修订。

### 2.1 最简记忆包

开写前必须先压成一份简短的“最简记忆包”，至少包含：

- 当前故事时间与当前直接危机
- 本章涉及人物的最新状态
- 本章相关伏笔
- 本章受哪条资源、关系、宗门、家族、时间约束

判断标准只有一条：

- 如果不知道这条信息，这一章就有较大概率写错，那就保留

和本章无关的设定、人物、远线伏笔不要硬塞进当前上下文。

### 3. 缺章纲时的处理

如果当前章没有明确章纲：

- 优先调用 `$jiazuxiuxian_novel-outline-planner` 先补大纲
- 如果用户就是要你直接写，也要先在心里生成一个简要场景提纲，再落正文

不要在毫无章纲和前文承接判断的情况下直接硬写。

### 3.1 续写与重写分流

如果用户说“续写第 N 章”：

- 先读旧稿
- 判断是接在旧稿末尾继续写，还是旧稿虽然存在但需要整章修
- 若旧稿已经形成完整章结构，但只差后半段，则优先接续
- 若旧稿方向、节奏、承接已经歪掉，不要硬续，改按重写处理

如果用户说“重写第 N 章”：

- 先读旧稿
- 必读前一章正文
- 若存在后一章，也必读后一章正文，避免重写后直接撞坏承接
- 重写后必须补查 `追踪/` 四件事：
  - `上下文.md`
  - `伏笔.md`
  - `时间线.md`
  - `角色状态.md`

### 4. 正文写法

正文默认服务于起点男频仙侠 / 家族修仙 / 势力经营取向。

保留这些特征：

- 第三人称近距离视角，核心感知紧贴当前主视角人物，默认以陆承前为主
- 章节标题格式统一为“第X章 章名”
- 单章体量默认控制在 4000 字左右，场景数量与推进密度要足以撑起这个篇幅，但不要靠重复解释和空转硬凑字数
- 开头常用明确时间、地点、场景锚点切入
- 开头第一拍优先落“谁在何处做什么”，不要先空写两三句气氛再把人物放进来
- 环境描写与内心复盘都要压量。能两三句交代清楚，就不要拖成整段；一旦开始讲道理，要尽快接回动作、对白或新信息
- 同一段里比喻尽量只留一个最有用的，不要一层动作再叠一层比喻、再补一层感慨
- 对话要体现试探、权衡、利害、家族位置与资源焦虑，而不是空泛喊口号
- 主角思考方式始终偏现实：先看资源、规矩、利益、执行条件，再看名分和感情
- 文气可以沉稳、克制，但可读性优先。先保证读者一遍能读顺，再谈文气和质感
- 优先用常用词、明白句、短中句。不要为了显得厚重、古雅、深刻，故意写绕、写满、写得像设定讲义
- 句式不要过分整齐，不要反复使用同一类排比、总结句、判断句
- 允许局部粗粝，宁可朴一点，也不要用力过猛地“写得像文学”
- 不要在动作或对白后再补一层作者点评
- 人情味主要落在称呼、停顿、递东西、回头、避视、收手、改口等具体反应上，不靠整段煽情解释
- 结尾尽量用动作、场景余波、半句话头收住，而不是每章都用概括性总结句收束
- 不要把人物都写得很会说话，配角语言要带身份、局限和生活气
- 家族修仙文的爽点主要来自：立规矩、保祖地、争灵田、稳药泉、拿矿脉、收客卿、养族人、补祖谱、抗宗门、扩势力
- 调查、对簿、查账、争地、族议、拍卖、守泉、守矿、宗门压价类场景，压字时可以砍味道句，但不能砍掉证据链、利益链和执行链上的关键一环

### 4.1 平台节奏校准

本 skill 默认服务于起点男频仙侠 / 家族修仙 / 经营流取向。

这不是可选增强，而是默认工作模式：

- 始终受 `references/qidian-editor-rules.md` 约束
- 始终参考 `references/qidian-opening-patterns.md`
- 即使用户没有额外说明，也默认按“更像起点、更有追读感、开头更能抓人”的方向处理

如果用户明确说“不要平台化”“不要起点感太重”，才允许适度收掉部分平台节奏，但仍不能放弃“每章有事务、冲突推动、章末牵引”这些基本要求。

执行原则：

- 第一章先让事发生，再补设定
- 第二章先把身份和关系落稳，再补规则
- 第三章必须让主角做出第一轮明确动作
- 每章至少要有：当章事务、当章阻力、部分兑现、章末牵引
- 主角必须在具体处境里使用自己的优势，不能只在脑中高明
- 文字优先流畅直白，不堆辞藻，不玩现代梗
- 追读感主要来自问题清楚、事务推进和章末余力，不来自句子故作深沉
- 仙侠家族读者更看重修为逻辑、资源逻辑、家族逻辑，不要靠无脑打脸推进
- 你的书仍保持沉稳、克制、现实的底色，不改成轻佻爽文、不改成插科打诨腔

### 4.2 仙侠家族文专项约束

- 不要写成历史文，不要借用东汉、官场、门禁、郎署、边报那套叙事逻辑
- 世界冲突要落在灵田、药泉、矿脉、坊市、宗门供奉、家族位序、婚盟、附庸、祖谱、灵脉上
- 修仙体系可以铺开，但不能把章节写成设定说明书
- 家族经营不能写成流水账。每一次盘资源、立规矩、调人手，都要服务于眼前危机
- 主角不能只是幕后分配，还要亲自出手、亲自破局、亲自承担代价
- 境界提升、产业扩张、族人培养、势力吞并要相互咬合，不能各写各的
- 既不能只剩爽点，也不能只剩沉重；要让读者持续看到“这一脉在站起来”

## 关于自然度与可读性

目标是写得更自然、更像连载稿，减少模板腔和明显机器感。

默认先满足一条：

- 普通网文读者第一次读时，基本不用回看句子，就能知道谁在做什么、这段话为什么要写

不要做的事：

- 不要为了“像小说”而每段都端着
- 不要为了“高级”而密集堆砌排比、长句、抽象判断
- 不要把简单意思故意写重、写古、写得半文半白
- 不要连续两到三段只靠气氛、心理、感慨撑住，而没有新动作、新信息、新对话
- 不要专门朝“规避检测”方向设计文本

要做的事：

- 多用常见词和直给句，先把意思说清
- 让信息落在人物、动作、场景和关系里
- 每一到两段至少给读者一个可见动作、对白或新信息
- 接受局部朴、局部粗、局部留白
- 保留已有正文已经形成的自然节奏，但文面默认向更顺口、更易读校正，而不是机械继承旧稿的沉重句法

## 落盘规则

默认写入：

- `正文/第NNN章_章名.md`

如果当前章原文件名已存在并带章名：

- 优先覆盖原文件

如果当前章只有 `第NNN章_待定.md`：

- 正文完成后可以按新章名重命名

写完正文后，至少检查：

- `追踪/上下文.md` 是否需要更新
- `追踪/伏笔.md` 是否新埋、推进或回收了条目
- `追踪/时间线.md` 是否需要推进
- `追踪/角色状态.md` 是否需要更新关键人物状态
