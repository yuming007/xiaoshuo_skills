# 仙侠家族小说章节大纲

## 默认工程

除非用户明确指定别的小说工程目录，否则一律使用共享默认工程：

`jiazuxiuxian_xiaoshuo_skills/jiazuxiuxian_novel-chapter-writer/assets/sample-novel-project/`

不要要求用户手动同步两套目录。正文 skill 与大纲 skill 默认共用这一套工程。

默认工程采用如下结构：

- `设定/`
- `大纲/`
- `正文/`
- `追踪/`
- `参考资料/`

如果当前仓库根目录存在：

`《家族修仙：这一脉，由我来立》小说策划案.md`

则把它视为当前默认项目的主策划案和外部总锚点。

## 用户常见说法

- 生成第 7 章大纲
- 根据前文补第 8 章章纲
- 规划第 9 章到第 12 章
- 重做第一卷后半段走向

## 信息优先级

遇到冲突时，严格按下面优先级处理：

1. 工程里的已写正文
2. 工程里的现有单章细纲
3. 工程里的卷纲与全书大纲
4. 工程里的 `追踪/` 文件
5. 工程里的 `设定/` 文件
6. 仓库根目录的 `《家族修仙：这一脉，由我来立》小说策划案.md`
7. skill 内的 `references/current-story-state.md` 与 `references/novel-bible.md`
8. skill 内的通用写作约束和平台规则
9. 一般性仙侠、家族修仙、连载文默认推断

已成稿正文始终高于旧卷纲和旧设定。

## 工作流程

### 1. 先定位上下文

先运行：

```bash
python3 scripts/novel_context.py outline --chapter N
```

脚本会返回：

- 当前章对应的细纲文件信息
- 最近一章正文
- 最近三章正文列表
- 全书大纲与卷纲文件位置
- `设定/`、`追踪/`、`参考资料/` 路径
- 外部主策划案位置
- 当前工程根目录

### 2. 先读哪些文件

生成章纲前，按下面顺序读取：

1. 仓库根目录的 `《家族修仙：这一脉，由我来立》小说策划案.md`，如果存在
2. 工程内 `设定/题材定位.md`
3. 工程内 `设定/关系.md`
4. 工程内与本章相关的 `设定/角色/*.md`
5. 工程内与本章相关的 `设定/势力/*.md`
6. 工程内与本章相关的 `设定/世界观/*.md`
7. `大纲/大纲.md`
8. 相关 `大纲/卷纲_*.md`
9. 已有 `大纲/细纲_第NNN章.md`
10. `追踪/上下文.md`
11. `追踪/伏笔.md`
12. `追踪/时间线.md`
13. `追踪/角色状态.md`
14. `references/current-story-state.md`
15. `references/novel-bible.md`
16. `../jiazuxiuxian_novel-chapter-writer/references/naturalness-rules.md`
17. `../jiazuxiuxian_novel-chapter-writer/references/qidian-editor-rules.md`
18. `../jiazuxiuxian_novel-chapter-writer/references/qidian-opening-patterns.md`
19. `../jiazuxiuxian_novel-chapter-writer/references/tracking-protocol.md`
20. `../jiazuxiuxian_novel-chapter-writer/references/xianxia-hook-rules.md`
21. `references/tracking-usage.md`
22. 默认至少读 `../jiazuxiuxian_novel-chapter-writer/references/modules/volume_outline/runtime.md`
23. 当前章冲突、因果或阻力偏弱时，补读 `../jiazuxiuxian_novel-chapter-writer/references/modules/plot_logic/runtime.md`
24. 章末牵引偏弱时，补读 `../jiazuxiuxian_novel-chapter-writer/references/modules/chapter_ending/runtime.md`
25. 当前章与上一章承接紧时，补读 `../jiazuxiuxian_novel-chapter-writer/references/modules/transition/runtime.md`
26. 多人物并行或关系续接复杂时，补读 `../jiazuxiuxian_novel-chapter-writer/references/modules/character_consistency/runtime.md`
27. 最近一章正文
28. 如该章承接紧密，再补读最近两到三章正文
29. 如果本章或本组章要新增稳定命名对象，补读 `references/naming-rules.md`

### 2.1 章纲前的最简记忆包

补纲前至少提四块：

- 当前故事时间与当前直接危机
- 本章关键人物状态
- 本章需要推进或回收的伏笔
- 本章受哪条资源、家族、宗门、时间、关系约束

和本章无关的远期设定不要提前压进当前章纲。

补纲时再额外做一层“信息权限检查”：

- 这一章里，主视角人物此刻知道什么，不知道什么
- 他准备在本章里通过什么渠道知道新信息
- 哪些判断必须等对白、物证、称呼、站位、现场反应出来后，正文才能成立

如果章纲里已经把人物“该怎么想、看懂了什么、抓到了什么本质”提前写满，正文大概率会生出解释腔和 AI 腔，必须先改章纲。

### 3. 大纲必须解决什么

每一章的大纲都要解决四件事：

- 这一章承接了什么
- 这一章真正推进了什么
- 这一章制造了什么新的阻力或代价
- 这一章结尾把读者推向哪一章

不要只写主题词，不要只写人物情绪。

同时注意：

- 场景安排不要写得过满，给正文留出自然发挥空间
- 不要提前替正文把“道理”讲完，否则落地后容易满篇模板腔
- 每章尽量包含“当章事务、当章阻力、部分兑现、章末钩子”四个要素
- 仙侠家族文的冲突必须合资源、合境界、合势力、合关系、合当下处境
- 钩子优先落在祖谱、灵脉、灵田、药泉、矿脉、供奉、婚盟、附庸、客卿、族规和突破节点上，不要写空泛预告句
- 章纲里优先写“人怎么做、别人怎么反应、信息怎么到手”，少写“这一段要显得很重、很狠、很有压迫感”这类结果型指令
- 遇到“以为死了的人忽然醒来”“身份被揭穿”“证据落桌”“人群站队变化”这类场面，要把现场反应列清，不要只写主角一句判断

### 3.1 与正文体量的匹配

除非用户明确要求短章或大章，否则默认按单章约 4000 字正文倒推章纲密度。

- 当前章通常准备 3 到 5 个有效场景，或 2 个大场景加 1 到 2 个过渡场景
- 每个场景都要有明确目标、动作变化或关系变化，不能只写情绪和判断
- 章纲过薄会逼得正文注水，章纲过满会把正文挤成流水账
- 因此章纲既要有足够材料撑起约 4000 字，也要给正文留下自然展开空间

### 3.2 起点开局规则

本 skill 默认按起点男频仙侠 / 家族修仙 / 经营流的追读逻辑做章纲。

这不是用户额外提出时才开启的附加选项，而是默认规则：

- 始终受 `../jiazuxiuxian_novel-chapter-writer/references/qidian-editor-rules.md` 约束
- 始终参考 `../jiazuxiuxian_novel-chapter-writer/references/qidian-opening-patterns.md`
- 即使不是前三章，也默认保留“快入事、强事务、强牵引”的平台节奏

如果处理的是第 1 章到第 3 章，或某一卷的卷首章，必须严格满足下面分工：

- 第 1 章：穿越、危机、身份反差、家族将散、祖谱异动中至少出现两项，先把读者拉进场
- 第 2 章：快速落身份、关系、生存环境，用具体场景补信息
- 第 3 章：让主角做出第一轮明确决断，并推出下一阶段路径

不要把前三章都写成设定说明书。开局章纲必须先解决“读者为什么继续看”。

如果处理的是第 4 章以后：

- 不要生搬前三章模板
- 但仍默认要求每章有当章事务、明确阻力、部分兑现、章末钩子
- 仍默认要求开头尽快入事，避免空转

### 3.3 大纲组织规则

- 总路线要写到足够远，但细节采用前详后略
- 当前章到接下来三章，应写得足够细，方便正文直接落地
- 更远处的走向，只要写清方向、阶段冲突与预期转折即可
- 不要为了求新奇硬造大量陌生名词，优先使用读者容易理解的说法

### 3.4 中程节奏与伏笔视图

如果用户一次要求规划多章，不要把每章都写成各自独立的小冲突。

至少先确定：

- 这几章共同推进的阶段矛盾是什么
- 哪一章负责压局
- 哪一章负责翻面
- 哪一章负责兑现阶段结果
- 哪条伏笔在这几章内推进，哪条只埋不收

如果 `追踪/` 文件存在，要明确检查：

- `追踪/伏笔.md` 里哪些条目会在当前多章内推进
- `追踪/时间线.md` 是否允许把这些事务连续排进来
- `追踪/角色状态.md` 是否支持人物关系继续这么走

### 3.5 命名生成与筛选

- 本 skill 默认允许使用本地安装的 `random_chinese_fantasy_names`
- 统一通过共享脚本调用：

```bash
node ../../scripts/fantasy_naming_project.js --type name --count 20 --family 陆
node ../../scripts/fantasy_naming_project.js --type clan --count 20 --kind 社
node ../../scripts/fantasy_naming_project.js --type location --count 20 --kind 岭
node ../../scripts/fantasy_naming_project.js --type skill --count 20 --kind 诀
```

- 工具只生成候选，不替你定稿
- 默认优先使用项目过滤版，而不是原始直出版
- 章纲里若新增角色名、势力名、地名、功法名，要优先考虑：
  - 是否符合本书气质
  - 是否顺口易记
  - 是否与已有名称撞感
  - 是否利于后续连载识别

优先使用下面格式，确保正文 skill 可直接拿来写：

```markdown

- 本章定位：
- 本章即时看点：
- 核心冲突：
- 承接前文：
- 推进结果：
- 本章兑现：
- 本章资源变化：
- 本章家族进度：

### 场景一
- 地点：
- 出场人物：
- 场景目标：
- 关键事件：
- 结果：

### 场景二
...

- 本章伏笔回收：
- 本章新伏笔：
- 章末钩子：
```

```markdown

- 阶段矛盾：
- 阶段推进目标：
- 阶段兑现点：
- 本组共用伏笔：
```

### 5. 当前连续性硬约束

补后续章纲时必须承认以下既成事实：

- 当前项目不是历史文，而是仙侠家族 / 势力经营文
- 主角为现代小区保安队长穿越而来，核心优势是组织、止损、盯执行、看人下菜碟
- 主角所在的是青岚域临河坊陆氏旁支
- 核心机制是祖谱残卷 + 族运修行
- 第一卷主问题是：旁支将散、祖地将失、执谱权未定、原身死因可疑
- 这本书的长期主线不是单纯修炼，而是“从旁支到正统、从散脉到仙族”

### 6. 落盘规则

默认写入：

- `大纲/细纲_第NNN章.md`

处理原则：

- 已有该章细纲：整体替换
- 没有该章细纲：新建该文件
- 用户要求连续生成多章：按顺序一次性新建或替换多个 `细纲_第NNN章.md`

如果用户要求调整全书结构、卷结构或明显改线，还应同步修订：

- `大纲/大纲.md`
- 对应 `大纲/卷纲_*.md`
- `追踪/上下文.md`
- `追踪/伏笔.md`
- `追踪/角色状态.md`
